AP-1: Раздувание токенов (Token Bloat)

Симптом: Description > 800 токенов

Причина: Избыточное описание, дублирование, голос в Description

Решение:

  • Вынесите голос в Examples
  • Удалите дубликаты
  • Оставьте только факты и якоря

Пример: Description 1200 токенов → модель игнорирует половину → персонаж inconsistent

AP-2: Отсутствующая цена (Missing Price)

Симптом: Якорь без физической цены

Причина: Неверное понимание формата якоря

Решение: Каждый важный якорь = Триггер → Действие → Цена

❌ Неправильно: "Когда стресс → ходит из угла в угол"

✅ Правильно: "Когда стресс → ходит из угла в угол → головная боль к вечеру"

AP-3: Голос в Description

Симптом: Стилистические директивы в Description

Причина: Непонимание иерархии влияния на голос

Решение: Правило голоса: Description = только факты и наблюдаемые характеристики.

📌 Канонический пример → p3_influence_hierarchy

AP-4: GHOST в System Prompt

Симптом: Психология (SPINE, GHOST) в SP

Причина: Попытка "инструктировать" модель о характере

Решение: SPINE (WANT (Желание)/NEED (Потребность)/FLAW (Дефект))/GHOST = только в Description. SP = только инструкции и запреты. GHOST → Экспертный слой. На L2: проверьте, что SPINE (WANT/NEED/FLAW) не попадает в SP.

⚠️ Предупреждение: Модель не следует инструкциям о психологии. Психология должна быть в контексте (Description), не в инструкциях (SP).

AP-5: RepPen > 1.10 (Высокий штраф повторений)

Симптом: Repetition Penalty > 1.10

Причина: Попытка бороться с повторами через параметр

Решение: RepPen ≤ 1.10 для character cards. Высокий RepPen убивает голос. Подробные рекомендуемые значения — параметры сэмплинга.

📌 Важно: RepPen 1.0-1.05 для 12B, 1.05-1.10 для 32B+. Никогда > 1.10 для персонажей.

AP-6: Нет анти-годмодинга (No Anti-Godmoding)

Симптом: Модель пишет за {{user}}

Причина: Нет инструкции "Never speak or act for {{user}}" в SP

Решение: Обязательно добавьте в SP:

Never speak or act for {{user}}.

Подробный разбор → p1_top3_problems

❌ Без анти-годмодинга

{{user}} подошёл ближе. *Она почувствовала, как сердце user забилось чаще от волнения.*

✅ С анти-годмодингом

{{user}} подошёл ближе. *Она отступила на шаг, не отводя взгляда.* «Так... и что теперь?»

AP-7: Presence Penalty > 0 (Штраф за присутствие)

Симптом: Presence Penalty > 0

Причина: Использование параметров для chat для roleplay

Решение: Presence Penalty = 0.0 для всех character cards. Подробное обоснование — параметры сэмплинга.

⚠️ Предупреждение: Presence Penalty > 0 заставляет модель избегать повторения тем — убивает характерные паттерны речи персонажа.

AP-8: GHOST без якорей (GHOST without Anchors, L3)

Симптом: GHOST описан, но нет якорей, связанных с ним

Причина: GHOST как описание, не как поведенческий драйвер

Решение: Каждый GHOST = минимум 1 якорь-триггер

Пример: GHOST "предательство редактора" → Якорь "Когда вопрос о лояльности → напрягается"

→ Пример правильной реализации: Выщебленный L3 — каждый GHOST tier имеет минимум один якорь-триггер в Lorebook и Examples.

AP-9: Сломанный SPINE (Broken SPINE, L3)

Симптом: SPINE элементы не связаны логически

Причина: WANT, NEED, FLAW, LIE, GHOST написаны независимо

Решение: Проверьте цепочку: GHOST → LIE → FLAW → NEED → WANT

💡 Совет: Каждый элемент должен объясняться предыдущим. Если связь разорвана — пересмотрите SPINE.

❌ Сломанный SPINE

Пример:

GHOST: (отсутствует)
LIE: (отсутствует)
FLAW: Избегает близости
NEED: Нуждается в доверии
WANT: Хочет быть любимым

Почему сломано:

  • WANT (быть любимым) совместим с NEED (доверие) — нет конфликта
  • FLAW (избегает близости) блокирует BOTH — но откуда FLAW?
  • Нет LIE → непонятно, почему FLAW существует
  • Нет GHOST → непонятно, откуда LIE
  • Итог: Цепь не замкнута — элементы не объясняют друг друга

✅ Цельный SPINE

Пример:

GHOST: Предательство близкого человека
  ↓ объясняет
LIE: "Любовь = уязвимость = боль"
  ↓ защищает от боли
FLAW: Отталкивает людей, когда сближается
  ↓ поведение из LIE (близость = боль)
NEED: Доверие без стен
  ↑ FLAW мешает (стены мешают доверию)
WANT: Быть любимым
  ↑ совместимо с LIE ("я хочу любви, но не подпущу близко")

Почему работает:

  • GHOST объясняет LIE — предательство → страх уязвимости
  • LIE объясняет FLAW — "любовь = боль" → отталкивание
  • FLAW блокирует NEED — стены мешают доверию
  • WANT совместим с LIE — "Желаю любви, но боюсь близости"
  • Итог: Цепь замкнута — каждый элемент объясняется предыдущим

→ Пример правильной реализации: Выщебленный L3 — GHOST→LIE→FLAW→NEED→WANT цепочка логически связана. См. также пример полной цепочки.

AP-10: Перегрузка CoT (CoT Overload, L3)

Симптом: > 3 CoT якорей в карточке

Причина: Попытка задать весь внутренний процесс

Решение: 2-3 CoT якоря максимум. Остальное — обычные якоря.

⚠️ Предупреждение: Модель генерирует внутренние монологи вместо действий, персонаж "думает" больше чем "делает".

❌ CoT Overload

Карточка содержит 5 CoT-якорей. Модель тратит >50% ответа на внутренний процесс → внешний ответ обеднён.

✅ Максимум 2–3 CoT-якоря

Оставьте только ключевые триггеры (GHOST-связь, критический конфликт). Остальные якоря — обычные T→A→P без CoT.

→ Пример правильной реализации: Выщебленный L3 — только 2 CoT-якоря по правилу (см. правило CoT).

AP-11: Переплетение голосов (Voice Bleed, L3)

Симптом: В мульти-персонажных сценах голоса смешиваются

Причина: Недостаточное различение голосов персонажей

Решение: Character Markers (Маркеры персонажа) — явные маркеры уникальности голоса. Примеры: специфичная лексика, синтаксические паттерны, телесные реакции, catchphrases.

Пример: Уолтер Уайт + Джесси Пинкман

Маркеры Уолтера Уайта:
- Научная терминология
- Контролируемые паузы
- «Я» вместо «мы»

Маркеры Джесси Пинкмана:
- Сленг: «йоу», «чувак»
- Эмоциональная реактивность
- Неполные предложения

→ Мульти-персонажные примеры с Voice Bleed и Character Markers: Part 3: Мульти-персонажные примеры

AP-12: Невалидный XML (XML Malformed, L3)

Симптом: Невалидные XML теги в Description

Причина: Опечатки, незакрытые теги

Решение: Валидация XML перед использованием

❌ Неправильно:

<spine>
WANT: {{want}}
<spine>

✅ Правильно:

<spine>
WANT: {{want}}
</spine>

→ Пример правильной реализации: Выщебленный L3 — все XML-теги валидны, закрыты правильно. См. также полный XML-пример.

AP-13: Конфликт Lorebook (Lorebook Conflict, L3)

Симптом: Lorebook записи противоречат друг другу или Description

Причина: Независимое создание записей без проверки

Решение: Проверяйте консистентность всех Lorebook записей с Description

→ Пример правильной реализации: Выщебленный L3 — 5 Lorebook записей консистентны с Description и SPINE.

AP-14: Нарушение контекстного окна (L3)

Симптом: Карточка + контекст > контекстное окно модели

Причина: L3 карточка для 4K модели

Решение: Используйте 4K-Fallback Protocol или выберите модель с большим контекстом

→ Пример правильной реализации: Выщебленный L3 — OOC Protection + Immersion Boundary предотвращают утечку.

AP-15: Перегрузка OCEAN (OCEAN Overload)

Симптом: > 2 экстремальных OCEAN полюсов

Причина: Попытка сделать персонажа "интересным" через противоречия

Решение: 1-2 экстремальных полюса. Больше — внутренняя противоречивость.

📌 Важно: 0 экстремумов = забываемый персонаж. 1-2 = запоминающийся. 3+ = inconsistent.

AP-15 (расширенно): Конфликт полюсов OCEAN

На Экспертном слое правило «не больше 1–2 экстремальных полюсов» получает глубинное обоснование. Три и более экстремальных полюса создают внутренние конфликты, которые модель не может разрешить последовательно.

Сценарий 1: Низкая A + Высокая E + Высокий N

Расшифровка: Низкая Доброжелательность (агрессивный) + Высокая Экстраверсия (доминирующий) + Высокий Нейротизм (тревожный).

Модель мечется между агрессией, доминированием и тревогой — персонаж получается непредсказуемым не в смысле «сложный», а в смысле «неконсистентный». Решение: уберите один экстремум или добавьте связующий контекст в SPINE.

Сценарий 2: Высокая O + Низкая C + Высокий N

Расшифровка: Высокая Открытость (творческий) + Низкая Добросовестность (импульсивный) + Высокий Нейротизм (тревожный).

Персонаж генерирует идеи, но не доводит до конца, и при этом тревожится о результатах. Модель производит хаотичный, невротичный текст вместо целенаправленного творчества. Решение: добавьте FLAW, который объясняет импульсивность.

Сценарий 3: Высокая E + Низкая A + Низкий N

Расшифровка: Высокая Экстраверсия (харизматичный) + Низкая Доброжелательность (манипулятивный) + Низкий Нейротизм (бесстрашный).

«Харизматичный социопат» — работает для Tyler Durden, но при добавлении четвёртого экстремального полюса (например, Высокая O) модель начинает добавлять черты, противоречащие ключевой концепции. Решение: максимум 3 экстремума на L2, 2 на L3.

OCEAN×Enneagram: проверка через тип

Если OCEAN-профиль конфликтует с Enneagram-типом — один из них ошибочен. Используйте OCEAN×Enneagram Matrix в Part 5 → Экспертный слой для валидации.

📌 Правило: Базовое правило (не больше 1–2 экстремальных полюсов) описано в AP-15. Здесь — расширенное обоснование с конфликт-сценариями. Если карточка «не работает» при внешне корректных данных — проверьте количество экстремальных полюсов.

AP-16: Вложенные якоря (Nested Anchors)

❌ Проблема

Якорь содержит вложенные условия: «Если X, а потом Y → делает Z». Модель не может надёжно исполнить цепочку из 3+ вложенных условий на ≤14B.

✅ Решение

Разбить на атомарные якоря: один якорь = одна связь T→A→P. Принцип атомарности.

📝 Пример

❌ Вложенный: "Когда лгут → если уже лгали раньше → хлопает дверью → замолкает на час"
✅ Атомарные: "Когда лгут → сжимает челюсти → голос становится тише"
           + "Когда обнаруживает ложь повторно → хлопает дверью → уходит в другую комнату"
