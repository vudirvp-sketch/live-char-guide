Базовые блоки карточки

Карточка персонажа — это документ, который задаёт поведение ИИ-модели при ролевой игре. Но думать о ней нужно как о поведенческом движке: каждый элемент должен производить наблюдаемое действие, а не украшать описание. Карточка состоит из четырёх блоков, каждый из которых выполняет свою функцию.

5 базовых правил

Эти правила обязательны для всех карточек, независимо от слоя. Нарушение любого из них приводит к ухудшению качества генерации.

📌 Фундаментальный принцип: LLM = Pattern Matcher, не Исполнитель правил. Директива в SP без паттерна в Examples нестабильна на моделях ≤14B (~40–60% следование директивам). Поэтому Examples — главный инструмент, а Description — логика поведения.

#ПравилоПочему важно
1OCEAN: 1-2 экстремальных полюса — значения <30 или >70Больше 2 — внутренняя противоречивость. 0 — персонаж забываемый.
2Цена обязательна — каждый FLAW/GHOST якорь имеет физическую реакциюБез цены модель не показывает уязвимость.
3Голос задаётся ТОЛЬКО в Examples и Greeting, никогда в DescriptionМодель игнорирует инструкции о голосе, но копирует паттерны из Examples.
4GHOST только в Description: GHOST (событие прошлого персонажа) — только в Description, никогда в System PromptПсихология в SP ломает поведение модели.
5Формат якоря: Триггер → Действие → ЦенаКаждый элемент обязательный. Пропуск — потеря качества.

Каждый блок имеет три уровня бюджета: минимальный (4K), стандартный (8K+) и максимальный (16K+). Превышение максимального значения — признак раздувания, ведущий к потере контекста.

Топ-3 критичные ошибки

Три самые частые ошибки начинающих, которые сильнее всего ломают качество карточки. Каждая разбирается подробно: симптом → причина → пошаговое исправление → пример до/после. Для краткого справочника по 5 проблемам см. Топ-5 проблем.

Ошибка 1: Персонаж пишет за {{user}}

Симптом: Модель описывает действия, мысли и реплики {{user}}. Персонаж отвечает на то, чего {{user}} не говорил. Диалог превращается в монолог от двух лиц.

Причина: В System Prompt нет анти-годмодинга. Модель «достраивает» диалог, потому что паттерн «вопрос→ответ» ей привычнее, чем ожидание ввода игрока.

Исправление:

  1. Добавьте в SP строку: "Never speak or act for {{user}}."
  2. Добавьте позитивную формулировку: "Only describe {{char}}'s actions and speech."
  3. Проверьте Examples: каждый пример должен заканчиваться репликой персонажа, а не {{user}}

❌ До

SP: "You are Elena, a cynical journalist."
Результат: модель пишет за {{user}} в 60–80% ответов.

✅ После

SP: "You are Elena, a cynical journalist. Never speak or act for {{user}}. Only describe Elena's actions and speech."
Результат: модель пишет за {{user}} менее чем в 5% ответов.

Ошибка 2: Голос не соответствует описанию

Симптом: Вы описали персонажа как «лаконичного и холодного», но модель отвечает многословно и эмоционально. Стилистика реплик не совпадает с Description.

Причина: Голос помещён в Description вместо Examples. Модель — Pattern Matcher: она копирует паттерны из Examples, но почти не следует стилистическим директивам в Description (0% влияние на 12B моделях).

Исправление:

  1. Удалите все стилистические инструкции из Description («говорит коротко», «использует сарказм»)
  2. Перенесите голос в Examples: покажите паттерн речи через 2 примера
  3. Убедитесь, что оба примера демонстрируют один и тот же голос

❌ До (голос в Description)

Description: "Маркус — детектив. Говорит отрывисто, командным тоном."
Examples: generic реплики без характера
Результат: модель игнорирует стиль, отвечает «нейтрально».

✅ После (голос в Examples)

Description: "Маркус — детектив. Якоря: ..."
Examples: "Где был в восемь?" / "Не тяни время. Мне нужны факты, не истории."
Результат: модель копирует отрывистый, командный паттерн (~85% следование).

Ошибка 3: Персонаж «слишком идеальный»

Симптом: Персонаж не проявляет уязвимость, не ошибается, реагирует «правильно» на все стимулы. Нет ощущения живого характера.

Причина: Якоря не имеют цены (Price). Без физической реакции якорь = инструкция без паттерна. Модель следует таким якорям лишь в 40–60% случаев.

Исправление:

  1. Проверьте каждый якорь: есть ли физическая цена в той же сцене?
  2. Замените абстрактные цены («чувствует вину») на наблюдаемые («голос становится тише, сжимает кулаки»)
  3. Убедитесь, что цена ограничивает персонажа, а не украшает

❌ До (якорь без цены)

"Когда кто-то лжёт → прищуривается, молчит"
Результат: модель иногда реагирует, иногда нет. Персонаж «ровный».

✅ После (якорь с ценой)

"Когда кто-то лжёт → прищуривается, отшучивается → напряжение в челюсти, голос становится тише"
Результат: модель показывает уязвимость в каждом 2–3 ответе при триггере.

Следующие шаги

Вы освоили минимальный слой L1 — этого достаточно для создания рабочей карточки. Если вы хотите углубить персонажа, два следующих слоя добавляют новые инструменты.

Глубокий слой (L2)

SPINE — фреймворк из 3 базовых элементов (WANT/NEED/FLAW), который связывает мотивацию персонажа с его поведением. Якоря перестают быть случайным набором правил и получают причину. OCEAN (Big Five) — инструмент проверки внутренней консистентности с интерактивным пентагоном. Enneagram — модель из 9 типов для выведения SPINE из типа личности. System Prompt получает CORE DIRECTIVES — единую систему из 5 директив. Также: Author's Note, Lorebook, Format Lock, анти-паттерны AP-1–AP-7 + AP-15 + AP-16.

  • Критерии перехода: карточка стабильна на L1, 2 Examples дают узнаваемый голос, 3–5 якорей покрывают основные ситуации
  • Время создания: ~30 минут (вместо ~15 на L1)
  • Токены: 800–1500 (вместо 400–800 на L1)

→ SPINE, FLAW-linked якоря, OCEAN/Enneagram: SPINE Framework

Экспертный слой (L3)

Если вы уже работаете с SPINE и хотите глубже — Экспертный слой добавляет LIE (ложную установку) и GHOST (событие прошлого), которые объясняют, почему персонаж ведёт себя именно так. Chain of Thought (CoT) — техника внутреннего процесса персонажа. XML-структура Description и API-блоки для Claude/GPT. Расширенные анти-паттерны AP-8–AP-14 и 12B-специфичная диагностика.

  • Критерии перехода: SPINE консистентен, 3 Examples показывают разные эмоции, FLAW-linked якоря работают, используете 32B+ или API модель
  • Время создания: ~60 минут
  • Токены: 1500+

→ LIE, GHOST, CoT, XML/API: Полный SPINE из 5 элементов

Заключение

Вы изучили базовые принципы создания карточки персонажа на минимальном слое L1. Ключевые правила, которые нельзя нарушать, собраны в 5 базовых правилах — запомните их, прежде чем двигаться дальше.

Что вы теперь умеете на L1:

  • Собрать рабочую карточку за 15 минут, заполнив 4 базовых блока (SP, Description, Examples, Greeting)
  • Написать 3–5 якорей в формате Триггер → Действие → Цена с обязательной физической ценой
  • Разместить голос персонажа только в Examples и Greeting, оставив Description для фактов и якорей
  • Соблюдать токен-бюджет: 400–800 токенов для L1 карточки
