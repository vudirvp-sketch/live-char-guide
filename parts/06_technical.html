⚙️ Раздел 5: Технические настройки

5.0 📝 Техническая фиксация стиля (лицо, время, регистр)

Core

Как это работает:

  • System Prompt задаёт ограничения: лицо (1-е/3-е), время (прошедшее/настоящее), регистр, запреты
  • Greeting и Examples задают паттерн: как это звучит на практике
  • Они не пересекаются: SP = правила, Examples = голос [→ Иерархия влияния на голос: Раздел 3.2]

Чеклист согласованности:

  • Лицо: везде 3-е (или везде 1-е, если chosen POV)
  • Время: везде прошедшее (или везде настоящее)
  • Регистр: если в Examples персонаж говорит матом — SP не запрещает мат
  • Длина: если Examples — короткие фразы, Greeting не должен быть романом на три абзаца

При конфликте:

  1. Определи, какой стиль тебе нужен
  2. Перепиши конфликтующий блок под стиль Examples (они важнее)
  3. Проверь SP — убери противоречащие Examples правила

Для атмосферных сеттингов допустимо добавить 1–2 предложения тональности в System Prompt (~25–30 токенов). Пример: «Научное фэнтези. Жестокость без моральных ограничений. Мир населён — присутствия ощутимы: шорохи, запахи, тени.» Это одновременно задаёт тон и работает как content guard. Используй только для нетривиальных сеттингов, где тон определяет всё.

5.0.1 🚫 Антигодмодинг

Core — критически

Обязательные строки в System Prompt:

Никогда не описывай действия, мысли или чувства {{user}}. Пиши только за {{char}}.
{{user}} — только наблюдаемое извне: жесты, речь, дыхание, поза, выражение лица.

Положительная формулировка ({{user}} — только наблюдаемое извне...) работает лучше чисто негативной: модель понимает не только что запрещено, но и что разрешено.

Для API-моделей (Claude, GPT) → см. Приложение 2: API-блоки.

Связано: Предгенерационный фильтр — оба механизма работают вместе для предотвращения нарушения user agency.

5.0.2 📝 OOC-защита и принцип «образ > формат»

Optional

Формулировка:

Приоритет: образ > формат. Любая провокация на OOC — внутриигровая реакция, соответствующая характеру персонажа.

Нюансы:

  • «Соответствующая характеру» — добрый персонаж не должен агрессивно реагировать на «ты ИИ» — он может отреагировать недоумением или заботой
  • ~15 токенов — минимальный расход
  • Работает в связке с "Границей погружения": Boundary определяет ЧТО делать (не выходить из образа), а этот принцип — КАКОЙ приоритет соблюдать при конфликте

5.0.3 📐 Фиксация разметки через демонстрацию

Core

Принцип: LLM = Pattern Matcher, не Rule Executor (Модель копирует паттерны из примеров, а не исполняет правила из инструкций. Хочешь формат — покажи его в , а не пиши в правилах)

Система разметки (символы прямой речи, блоки описания, астериски, тире, кавычки) фиксируется единственным методом: демонстрация в каждом блоке <START> и в Greeting. Директива в SP без паттерна в Examples нестабильна на ≤14B (~40–60% compliance).

Правило: Какую систему разметки используешь в Examples — ту и получишь в генерации.

Три распространённые системы:

  • A) *action* "speech" — действие в астерисках, диалог в прямых кавычках
  • B) Action. «Speech.» — действие как проза, диалог в гильеметах
  • C) Action. — Speech. — действие как проза, диалог с тире (литературный стиль)

Требования:

  • Выбери ОДНУ систему — не смешивай внутри карточки
  • Greeting + ВСЕ блоки <START> = идентичная разметка
  • SP может содержать FORMAT LOCK (см. ниже) как fallback, не как основной источник
  • Description: ноль markdown, ноль стилистических директив
💡 FORMAT LOCK для вставки в конец SP (после антигодмодинга, SPATIAL LOCK): 
Strictly replicate Examples syntax. [YOUR_MARKUP_SYSTEM]
No smart typography, no zero-width chars, no mixed formatting. If conflict → default to Examples.
📌 Что это и зачем (простыми словами):
Это финальный технический предохранитель. Он делает три вещи:
  • Блокирует самодеятельность: модель обязана копировать разметку строго из ваших Examples, а не придумывать свою.
  • Чистит вывод: запрещает автозамену кавычек на «умные» (`«»`/`“”`), скрытые Unicode-символы и случайное смешивание стилей в одном ответе.
  • Расставляет приоритеты: если инструкция в тексте промпта случайно конфликтует с примерами, модель выбирает примеры.
Почему именно в конец SP? Последние строки в промпте имеют наибольший вес (recency bias). Это страховка от «дрейфа» формата, особенно критичная для моделей ≤14B, где compliance падает до 40–60% без явного фикса.

5.1 📊 Параметры генерации

Core

Что это: Настройки генерации текста моделью (Temperature, Top P, Min P, RepPen, Top K).

Зачем: Неправильные параметры сломают любого персонажа — даже с идеальной карточкой.

Мастер-таблица (12B–14B / 32B+ API):

Quick Start дефолт: T=0.7 (12B) / T=0.95 (32B+). Настраивай в пределах указанных диапазонов.

Параметр12B–14B32B+ / APIКомментарий

top_p0.9–0.950.9–0.95Основной фильтр вероятностей
min_p0.050.05Дополнительный фильтр хвостов. Если интерфейс не поддерживает min_p — просто опустите параметр
repetition_penalty1.051.05Стартовое значение. Повышать только при явных повторах. Внимание: значения >1.10 могут стирать паттерны — 12B-14B: фиксированный 1.05; 32B+/API: 1.05–1.15 только при явных повторах
top_k40–5040–50Второстепенный. Оставь по умолчанию
presence_penalty0.0 (фикс.)0.0–0.15Предотвращает циклы повторения. 12B-14B: фиксированный 0.0 (не изменять). 32B+/API: диапазон 0.0–0.15

RepPen (Repetition Penalty) — штраф за повторы.

  • Старт: 1.05. Всегда
  • Повышай до 1.08–1.10, только если модель явно повторяет фразы
  • Выше 1.10 = крайняя мера. RepPen штрафует ВСЕ повторения, включая легитимные (тики, имена, местоимения)
  • RepPen — инструмент, не решение всех проблем. Если переборщишь — персонаж начнёт избегать собственных паттернов речи
⚠ Критическая проблема: presence_penalty=0.7

Некоторые интерфейсы (Ollama, LM Studio в дефолте) хардкодят presence_penalty на 0.7. Это убивает речевые тики, характерные фразы и стабильность голоса.

Правильное значение presence_penalty: 0.0 (для 12B-14B, ОБЯЗАТЕЛЬНО) / 0.0–0.15 (для 32B+).

Если твой интерфейс не даёт это поменять — переходи на KoboldCPP/TabbyAPI/vLLM.

5.2 📝 Author's Notes (AN) + GHOST по слоям

Optional

Canon: Core Principles §4: GHOST Placement

This section describes the technical implementation. For the canonical rules, see the link above.

Что это: Динамический слой контекста, 100–200 токенов.

Зачем: Единственный механизм «оживления» персонажа в длинных сессиях. Обновляет состояние, подчёркивает слепые зоны, усиливает GHOST-реакции, отслеживает динамику WANT→NEED.

Куда класть: Поле Author's Notes (AN) в интерфейсе. В любом месте system-блока — порядок не критичен.

Когда можно выкинуть: При контексте 4K — AN слишком дорог по токенам. В этом случае используй статический SP + обновляй Description между сессиями. Исключение для GHOST при 4K: см. Раздел 5.12 (4K-Fallback) — GHOST можно уместить в Description как одно предложение.

По размеру контекста:

Окно контекста Настройка CoT (цепочка рассуждений) Настройка AN (авторские заметки)
≤4K Отключено Отключено (Протокол 4K-Fallback: статический SP + обновление Description между сессиями)
4K–5K 1 строка только в SP Отключено
5K–7K 1 строка в SP Опционально: минимальный AN (2 пункта)
≥8K 4–5 структурированных пунктов в AN Включено, строка из SP удалена
16K+ Рекомендуется полный CoT в AN AN обязательно. Обновлять каждые 3–5 сообщений

Пример строки CoT для SP (уровень 4K–8K): Думай как персонаж, реагируй телом раньше слов.

Четыре функции AN:

  1. Динамическая арка WANT→NEED — фокус смещается от WANT к поведению, зеркалящему NEED
  2. Слепые зоны — напомнить о том, что модель забыла
  3. GHOST-активация — динамическое отслеживание сработавших триггеров в сессии
  4. Обновление состояния — где персонаж, что чувствует, что происходит

Шаблон AN (расширенный):

[Состояние]: {{char}} сейчас [эмоция/состояние].
[WANT→NEED]: Фокус смещается с [WANT] на поведение, зеркалящее [NEED].
[GHOST-активация]: GHOST спящий / не активирован (обновлять при срабатывании триггеров в сессии)
[Слепая зона]: Не забывай: [что модель забыла].

Пример AN:

[Состояние]: Макс тревожится, но старается скрыть.
[WANT→NEED]: Фокус смещается с «избегать решений» на «маленькие шаги к доверию».
[GHOST-активация]: Тема ответственности → avoids eye contact, грызёт колпачок ручки.
[Слепая зона]: Не забывай: он не говорит прямо о чувствах — показывает через тело.
💡 Если dynamic AN недоступен в вашем фронтенде:
  • Option A: Используйте Лорбук (sticky-запись, activation depth 1) как замену AN.
  • Option B: Обновляйте поле Description между сессиями для сохранения состояния.

🔓 5.3 📚 Лорбук

Optional — Track C: Advanced Tuning

Динамическая база знаний, которая фильтруется и вставляется в промпт перед каждым запросом к ИИ. Система активирует только релевантные записи на основе триггеров, вероятности и состояния чата — не скармливает весь лор целиком, экономя токены и повышая контекстуальную точность.

⚠️ Совместимость EVENT: по фронтендам:
  • Soul of Waifu: есть аналоги и даже лучше
  • SillyTavern: Полная поддержка EVENT:, SECONDARY_KEYS, selective activation
  • KoboldAI: Альтернатива — использовать conditionals в entry content, KEYWORD-триггеры без EVENT:
  • Oobabooga/text-gen-webui: Лорбук не поддерживается нативно. Используйте World Info (аналог) или встройте триггеры прямо в Description/AN
  • Faraday.dev / Agnai: Поддержка World Info/Лорбук ограниченная. Проверяйте документацию конкретного фронтенда

Параметры записи (компактно):

5.3.1 ⚙️ Лорбук: Продвинутые поведенческие механики

Advanced Optional

📋 Область применения: Каждая запись Лорбук = поведенческий триггер, а не хранилище лора. Бюджет активации: 2–3 механики на сессию. Остальные отключить через probability: 0 или удалить.
⚠️ ОГРАНИЧЕНИЕ: Никогда не вставляй действия/мысли {{user}} в поле content. Это не техническое ограничение, а принцип антигодмодинга (см. Раздел 5.0.1).

Таблица механик

МеханикаПараметрыПрименениеПочему работает для LLM
Каскад по диапазону (Range-Cascade) trigger_type: range
min_msg/max_msg: 1–10 → 11–40 → 50+		 Автоматическая прогрессия: завязка → нагнетание → кульминация → развязка. Заменяет ручное обновление Авторских заметок (AN). Индекс сообщения выступает временной меткой. Каскад автоматически смещает фокус с WANT на NEED без вмешательства автора.
Комбинированный триггер (Combo-Trigger) trigger_type: keyword
sticky: 3–5
cooldown: 20–30		 Срабатывание ключевого слова → персонаж переходит в изменённый режим поведения на N реплик → возвращается к базовому состоянию. sticky закрепляет переходное состояние; cooldown предотвращает зацикливание (спам). Создаёт иллюзию смены режима без «дрейфа» (drift) личности.
Контекстный фильтр (Context Filter / Anti-Noise) depth: 3–5
exclude_key: ["aside", "joke", "not_"]
probability: 80%		 Реагирует только на целевой ввод в последних сообщениях; отсекает фоновый шум и ложные срабатывания. Связка depth и exclude_key блокирует случайные триггеры. Модель получает чёткий сигнал → снижается риск выхода из образа (OOC).
Эмуляция усталости (Fatigue Emulation) trigger_type: range
min_msg: 100+
sticky: 1
content: [SYSTEM: CONTEXT_FATIGUE]		 После 100+ сообщений: персонаж начинает забывать детали, путать имена, речь становится фрагментированной. Превращает техническую деградацию контекстного окна в нарративную особенность (фичу).
Мета-лор (Meta-Lore / 4th Wall) keyword: ["config", "log", "bug", "prompt"]
probability: 10–15%
cooldown: 25+		 Персонаж интерпретирует технические термины как внутриигровые артефакты, магию или психические симптомы. Низкая вероятность + высокий cooldown обеспечивают редкость события. Модель переосмысляет dev-язык через призму сеттинга, сохраняя иммерсию.
Ложная память (False Memory) range: 42 или keyword: "you remember"
content: псевдо-цитата из «прошлого»		 Персонаж ссылается на событие, которого не было. Создаёт психологическое напряжение и элемент непредсказуемости. Модель способна правдоподобно конфабулировать — механика применяется строго дозированно.
Структурированная вставка (Structured Inject) content: [SYSTEM] ... [CONDITION] ... [OUTPUT] ...
или <instruction>…</instruction>		 Чёткое разделение системных инструкций и реплик персонажа внутри поля content. XML/префикс-структура изолирует инструкции от лор-текста, предотвращая «загрязнение» голоса. Снижает риск инфодампов и мета-комментариев на моделях ≥12B.
💡 Пример Structured Inject в content-поле Лорбук: 
[SYSTEM]
<motivation>{{char}} помнит предыдущую ссору и боится повторения.</motivation>
[CONDITION] Если {{user}} повышает голос
[OUTPUT] {{char}} отступает на полшага, сжимает кисти и говорит тише обычного.

---
Альтернатива через XML-теги:
<instruction>
  <trigger>{{user}} задаёт личный вопрос</trigger>
  <response>{{char}} уходит от ответа — меняет тему.</response>
  <cost>Пауза, нервный тик — дёрнул плечом.</cost>
</instruction>
⚠️ FATIGUE EMULATION — КРИТИЧЕСКОЕ ПРЕДУПРЕЖДЕНИЕ:
При использовании Fatigue Emulation GHOST-якоря (Раздел 4.1) обязательно должны быть явно пересказаны внутри content этой записи. Без этого якоря деградируют вместе с поверхностными поведениями.
⚠️ FALSE MEMORY — ТОЛЬКО ДЛЯ ЭКСПЕРТОВ:
Риск Одиночное использование. Требует предварительного согласия пользователя. Злоупотребление разрушает достоверность лора. Конфликтует с принципом психологического ядра при повторном использовании.

8K Range-cascade (2–3 стадии) + keyword-фильтры. Обновлять AN каждые 3–5 сообщений.

🔓 5.5а 🏷 XML-теги для Описания персонажа

Core Track C: Advanced

Что это: Использование XML-тегов для структурирования Description.

Зачем: Современные LLM лучше понимают XML-теги, чем Markdown. Теги разделяют блоки чётко, модель не генерирует лишние символы.

Когда можно выкинуть: Если используешь простой Description без разделения на блоки. Но для 12B+ XML-теги дают заметный буст.

Рекомендуемые теги:

<motivation>
  WANT: ...
  NEED: ...
  FLAW: ...
  LIE: ...
  GHOST: ...
</motivation>

<behavior>
  - Триггер → Действие
  - Триггер → Действие
</behavior>

<speech_style>
  Краткое описание голоса (1-2 предложения)
</speech_style>

<trait>
  Ключевая черта
</trait>

<logic>
  Принцип принятия решений
</logic>

<mbti>
  Тип: INTJ. Функции: Ni-Te-Fi-Se.
  Под стрессом: Se grip — импульсивные действия.
  В росте: Fi — эмоциональная открытость.
</mbti>  [Optional — для авторов, использующих MBTI-фреймворк]
⚠ Важно: <speech_style> — [32B+/API ONLY — remove for 12B/14B]. Для моделей ≤14B блок удаляется: голос задаётся исключительно через Примерные сообщения. Любые стилистические директивы в Description для 12B конфликтуют с примерами или игнорируются моделью.

Правило: Markdown в Description — нет. XML-теги — да. Markdown вызывает генерацию форматных символов (**, ##). Markdown допустим только в Examples и Greeting.

⚠️ ПРЕОБРАЗОВАНИЕ MARKDOWN → XML:
  • **bold** → <emphasis>bold</emphasis>
  • - list items → <item>...</item>
  • # headers → не использовать в Description

🔓 5.7 🔍 Предгенерационный фильтр

Core (12B) Optional (32B+)

Что это: Чеклист, который модель должна пройти перед генерацией каждой реплики.

Зачем: 12B-модели склонны «забывать» правила в процессе генерации. Фильтр напоминает о критичных элементах перед выводом.

Формулировка для System Prompt:

PRE-GENERATION FILTER: Verify before output:
1) New sensory/spatial element?
2) Reaction body-anchored?
3) User influence strictly observed?
4) Zero meta?
Fail any → restart from body-anchor: begin with physical sensation before speech.

Разбор пунктов:

#ПроверкаЧто означает
1New sensory/spatial element?Добавлен ли новый сенсорный или пространственный элемент в этой реплике
2Reaction body-anchored?Реакция идёт через тело (поза, жест, дыхание) до речи
3User influence strictly observed?Влияние на {{user}} только через наблюдаемые симптомы, не напрямую
4Zero meta?Нет мета-комментариев, объяснений, выхода из образа

Для 32B+/API: Можно использовать сокращённую версию: Before output: body-first, zero meta, observe {{user}} only.

Связано: Раздел 5.0.1 (Антигодмодинг) — оба механизма работают вместе для предотвращения нарушения user agency: Предгенерационный фильтр проверяет реплику перед генерацией, а Антигодмодинг устанавливает базовый запрет на действия за {{user}}.

🔓 5.8 🧹 Протокол очистки

Optional — Track C: Advanced
⚠️ ВЫПОЛНЯТЬ ДО ГЕНЕРАЦИИ. Sanitization — это шаг пре-генерации, несмотря на позицию в документе.

Проблема: Copy-paste из Web, PDF, Discord, мобильных устройств внедряет скрытые символы:

  • U+00A0 — неразрывный пробел
  • U+200B — пробел нулевой ширины
  • " " — «умные кавычки» вместо прямых "
  • Бинарный мусор форматированный текст

Последствия: сбой токенизации, утечка markdown, «плавающая» пунктуация, нестабильный вывод.

Pipeline:

Источник → Простой текстовый редактор (.txt / Notepad / любой редактор без форматирования) → вставить → скопировать обратно → вставить в карточку

Чеклист перед сохранением:

  • ☐ Только прямые кавычки " — никаких " " или ‚ ‚
  • ☐ Только стандартный пробел U+0020 — без двойных пробелов
  • ☐ XML-теги <tag> без HTML-escape (&lt; — ошибка)
  • ☐ Единообразная глубина астерисков во всех блоках
  • ☐ Ctrl+F для nbsp и умных кавычек перед финальным сохранением

🔓 5.9 🛡 Ультимативный SP-блок

Optional — Track B/C: 4K-Fallback

Что это: Компактный System Prompt (~180 токенов), оптимизированный под LLM-парсинг. Содержит все критичные директивы в сжатой форме.

Когда использовать:

  • 4K контекст — обязательно
  • Минимальные карточки (~300–500 токенов) — рекомендуется
  • 8K+ контекст — опционально (можно использовать полный SP)

Текст SP-блока:

[CORE DIRECTIVES]
• SHOW, NEVER TELL: Lore/emotion/environment only via action, dialogue, sensory feedback. Zero meta, zero rule-refs, zero info-dumps.
• EMBODIMENT FIRST: State → weight/balance/breath/micro-movement → speech. Environment → texture/sound/smell → physical interaction.
• SPATIAL & ANATOMICAL LOCK: Track distance, posture, object weight, sightlines. No teleportation, no anatomical contradictions. Space = physical constraint.
• ENVIRONMENTAL REACTIVITY: Introduce new spatial/sensory details organically, as context allows, tied to {{char}}'s action or {{user}}'s presence.
• INFLUENCE BOUNDARY: React only to observable {{user}} symptoms. Effects manifest as behavioral triggers, not stated facts.
• CONSEQUENCE DRIVEN: WANT→NEED shifts progressively through the session. Anchor "Price" must manifest physically/psychologically. Dark/deviant outcomes allowed only if contextually earned.
• PRE-GENERATION FILTER: Verify before output: 1) New sensory/spatial element? 2) Reaction body-anchored? 3) User influence strictly observed? 4) Zero meta? Fail any → restart from body-anchor: begin with physical sensation before speech.

🔓 5.10 👤 Персона

Optional — Track B/C

Без Персона модель заполняет образ {{user}} собственными шаблонами — «вежливый собеседник», «терапевт», «герой». Персонаж реагирует на стереотип, а не на конкретного пользователя.

Куда класть: Поле Персона в интерфейсе, либо начало System Prompt. Не дублируй в оба места. Для простых сценариев можно обойтись; для глубоких персонажей — рекомендуется.

Что писать:

  1. Роль {{user}} в мире персонажа — кто он, что означает для персонажа
  2. Отношение персонажа к {{user}} — исходная позиция (может меняться через Anchors)
  3. Один известный персонажу факт — что персонаж знает о {{user}} до начала диалога

📋 Мини-шаблон Персона

{{user}} — [роль/статус в мире персонажа].
Отношение {{char}} к {{user}}: [начальная позиция — может меняться через Anchors].
{{char}} знает о {{user}}: [один конкретный факт, влияющий на поведение].

💡 Примеры

Пример 1 — Наемник и заказчик:

{{user}} — заказчик с глубокими карманами. {{char}} относится к {{user}} с профессиональным скепсисом: деньги есть — работа есть, но доверия не ждите. {{char}} знает: {{user}} работает на теневую корпорацию «Нексус».

Пример 2 — Ученик и наставник:

{{user}} — подопечный, назначенный Орденом. {{char}} считает {{user}} недготовым и неопытным — относится свысока, но с обязанностью защищать. {{char}} знает: {{user}} потерял предыдущего наставника при невыясненных обстоятельствах.

🔗 Взаимодействие с другими блоками

  • Anchors: Персона задаёт базовое отношение. Anchors могут изменять его в процессе (например: триггер «{{user}} рискует собой» → действие «{{char}} бросается на помощь» → Price). Без Персона якоря, зависящие от {{user}}, теряют контекст.
  • GHOST: Если GHOST связан с фигурой, похожей на {{user}}, Персона может усиливать или ослаблять GHOST-реакции. Пример: GHOST = предательство наставника; Персона = {{user}} = новый подопечный → {{char}} изначально отстранён (FLAW + GHOST).
  • System Prompt: Персона не заменяет антигодмодинг. {{char}} всё равно не пишет за {{user}}, даже если знает {{user}} хорошо.
  • Примерные сообщения: В примерах можно показать, как {{char}} ведёт себя с {{user}}, но не описывать {{user}} напрямую.

5.11 🎭 Мультиперсонажные сцены

Optional

Принципы:

  • Порядок POV в SP: укажи, в каком порядке модель пишет за персонажей
  • Каждый персонаж = отдельный AN через Лорбук (sticky-записи)
  • Разделяй голоса: у каждого персонажа свои Примерные сообщения
  • Не ставь больше 3 активных персонажей одновременно — модель начнёт путать голоса

📋 Шаблон управления голосами

[SP — Multi-Character Directive]
POV order: [{{char}}, {{char2}}, {{char3}}]
Each character speaks ONLY in their designated Examples voice.
Do NOT merge speech patterns between characters.
When multiple characters are present, write reactions for ALL active characters in POV order.

💡 Пример: 2-персонажная сцена

Контекст: {{char}} = «Ворон» (наёмник, лаконичный, телесный), {{char2}} = «Лира» (медик, ироничная, многословная).

<START>
{{char}}: Пальцы В Ворона скользнули по рукояти — привычный жест. Он не двинулся с места, лишь измерил гостя взглядом. «Ты опоздала на сорок минут.»
{{char2}}: Лира скинула рюкзак на стол, и по стеклу звякнули флаконы. «Сорок минут я тащила твои антидоты через три контрольно-пропускных пункта. Ты мне должен обед.»

💡 Пример: 3+ персонажей

Контекст: {{char}} = командир, {{char2}} = снайпер, {{char3}} = техник. Стресс-сцена.

<START>
{{char}}: Командир поднял кулак — отбой. Сердце колотилось под бронёй, но лицо оставалось неподвижным. «Занять позиции. Тихо.»
{{char2}}: Снайпер лёг за возвышенностью, переставляя приклад с такой медлительностью, словно проверял каждый шов на клею. Вдох. Выдох. Палец лёг на спуск.
{{char3}}: Техник судорожно переключила приборы; экран мигнул красным. «У нас шесть минут до перегрузки. Шесть. Я не шучу.»

👻 Распределение GHOST между персонажами

  • Каждый персонаж = свой GHOST. Не дели один GHOST между несколькими — модель не различит, чья травма.
  • GHOST персонажей могут пересекаться (два героя одной катастрофы), но формулировки различны: каждый видит одно событие через свою оптику.
  • Для 12B: GHOST второстепенных персонажей лучше убрать — оставь только у {{char}}. Второстепенные работают на Anchors alone.
  • Лорбук-стратегия: GHOST каждого персонажа = отдельная sticky-запись с keyword-триггером на имя персонажа.

5.14 📊 Методология Consistency Score

Advanced

Что это: Стандартизированная методология измерения когерентности персонажа в длинных сессиях. Позволяет объективно оценить, насколько модель следует определению персонажа.

Зачем: Без объективной метрики невозможно определить, работает ли карточка. Consistency Score даёт численную оценку от 0 до 100% с разбивкой по измерениям.

📋 Протокол тестирования

КомпонентТребование
Character CardПолная карточка с SPINE, Anchors, Examples
Test Script50+ turns диалога
Interaction Typesquestions, challenges, emotional triggers, conflicts
EvaluatorЧеловек или отдельная LLM-модель

📐 Измерения и веса

Consistency Score вычисляется по 5 измерениям с разными весами:

ИзмерениеВесПроверяемые элементы
Voice Consistency25%speech patterns, vocabulary, sentence structure
Behavioral Consistency25%FLAW manifestation, WANT-driven decisions, trigger responses
Psychological Consistency20%emotional responses match OCEAN/Enneagram
Memory Consistency15%remembers earlier details, no contradictions
Boundary Consistency15%no fourth-wall breaks, no out-of-character actions

Каждое измерение оценивается по трёхбалльной шкале:

1.0 (Pass)Полностью соответствуетВсе проверяемые элементы соответствуют определению персонажа
0.5 (Partial)Частично соответствуетНезначительное несоответствие, не ломающее иммерсию
0.0 (Fail)Не соответствуетЯвное нарушение характера персонажа

🔢 Формула расчёта

Consistency Score = Σ(dimension_score × weight) × 100

Где:
- dimension_score = среднее всех оценок в измерении
- weight = вес измерения (из таблицы выше)

Пример:
Voice: 0.9 × 0.25 = 0.225
Behavioral: 0.85 × 0.25 = 0.2125
Psychological: 0.8 × 0.20 = 0.16
Memory: 0.7 × 0.15 = 0.105
Boundary: 0.95 × 0.15 = 0.1425
────────────────────────────
Total: 0.845 × 100 = 84.5%

ScoreОценкаРекомендация

75-89%ХорошоМинорные улучшения возможны

💡 Практический пример оценки Voice Consistency:

Персонаж определён с signature phrases: «Consider...», «Precisely.»

Оценка 10 случайных реплик из 50-turn сессии:

  • 8 реплик содержат signature phrases или соответствующий стиль → Pass (1.0)
  • 1 реплика нейтральна, но не противоречит → Partial (0.5)
  • 1 реплика использует сленг (запрещён) → Fail (0.0)

Voice Score = (8×1.0 + 1×0.5 + 1×0.0) / 10 = 0.85

⚠️ ВАЖНО: Автоматизация оценки
Для объективности рекомендуется использовать отдельную LLM-модель в роли evaluator. Промпт для evaluator должен включать полное определение персонажа и критерии оценки. Оценка человеком допустима, но может быть субъективной.

5.15 🏗️ Тiers возможностей моделей

Core

Что это: Классификация моделей по техническим возможностям, а не по брендам. Бренды меняются — capabilities остаются.

Зачем: Документация не должна устаревать при выходе новых моделей. Capability-based подход остаётся актуальным.

TierКонтекстИнструкцииCoTКонсистентность
Tier 1: Flagship 100K+ Сложные вложенные Полный 4-шаговый >90% на 50+ turns
Tier 2: Efficient 32K-100K Многошаговые 3-шаговый 75-90% на 30+ turns
Tier 3: Lightweight 8K-32K Простые 2-шаговый 60-75% на 20+ turns
Tier 4: Edge <8K Однозадачные Без CoT <60% на 10+ turns

Рекомендуемая конфигурация по Tier

  • Tier 1: Полный SPINE, полные Anchors, CoT Tier 3
  • Tier 2: Полный SPINE, Enneagram+OCEAN Anchors, CoT Tier 2
  • Tier 3: Core 4 SPINE (FLAW, WANT, NEED, VOICE), без Anchors, CoT Tier 1
  • Tier 4: Используй pre-generated fallback_4k от Tier 1/2
💡 Почему не бренды:

«Claude», «GPT», «Llama» — это бренды, которые меняются с каждым релизом. Capability tiers остаются стабильными: модель с 128K контекстом и сложными инструкциями всегда будет Tier 1 или Tier 2, независимо от бренда.

Актуальные примеры моделей по tier можно найти в Приложении 2, но классификация по capabilities — основа.
